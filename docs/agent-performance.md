# Agent パフォーマンス監視ダッシュボード

Copilot PR Reviewer（およびその他の LLM Agent）のパフォーマンス・品質を追跡するファイル。

## 概要

Agent は PR レビュー・コード品質評価を提供します。その出力品質と効率を定量的に測定し、改善に活かします。

---

## 📊 PR ごとのレビューメトリクス

| PR # | Issue | タイトル | コメント数 | Critical | Important | Nice-to-have | 対応率 | 品質スコア | 実装コスト | 備考 |
|------|-------|---------|----------|----------|-----------|-------------|-------|-----------|----------|------|
| 29 | #28 | `/admin/settings` ページ実装 | 6 | 3 | 2 | 1 | 100% | ⭐⭐⭐⭐⭐ | 中 | 全コメント対応、高精度レビュー |
| 27 | #10 | 店舗設定 API（バックエンド） | 11 | 4 | 4 | 3 | 36% | ⭐⭐⭐ | 低 | 部分対応：Critical/Important のみ対応 |

### 列の定義

- **コメント数**: 総レビューコメント数
- **Critical**: 機能・データ整合性・セキュリティの問題
- **Important**: 型安全性・エラーハンドリング・UX改善
- **Nice-to-have**: スタイル統一・軽微な改善
- **対応率**: (対応したコメント数 / 総コメント数) × 100%
- **品質スコア**: Agent レビューの妥当性
  - ⭐⭐⭐⭐⭐: 指摘が正確で実装に直結、なおかつ適切
  - ⭐⭐⭐⭐: 指摘が正確、対応価値あり
  - ⭐⭐⭐: 指摘が大部分正確、一部判定に議論の余地
  - ⭐⭐: 指摘に重複・軽微なものが多い
  - ⭐: 誤指摘が多い、または価値が低い
- **実装コスト**: 対応に必要な工数
  - 低: 簡単な修正で済む
  - 中: テスト追加など中程度の対応が必要
  - 高: 大幅な設計変更が必要

## 🎯 品質基準・目標

### 対応率の目安

| 分類 | 目標 | 判定基準 |
|-----|------|---------|
| **Critical** | 100% | セキュリティ・機能問題は全て対応 |
| **Important** | 80% 以上 | 品質に影響する指摘はほぼ全て対応 |
| **Nice-to-have** | 50% 以上 | 効率判断で選別可 |

### 品質スコア目安

| スコア | 意味 | アクション |
|--------|------|-----------|
| ⭐⭐⭐⭐⭐ | 高精度レビュー | 次の PR でも信頼して対応 |
| ⭐⭐⭐⭐ | 通常品質 | 標準的な対応フロー |
| ⭐⭐⭐ | 要確認 | コメント内容を人間が精査 |
| ⭐⭐ 以下 | 低品質 | プロンプト・Agent 設定の改善検討 |

## 🔧 対応判定フロー

```
PR レビューコメント受け取り
  ↓
【ステップ 1】コメント分析
  - 🔴 Critical 分類
  - 🟡 Important 分類
  - 🟡 Nice-to-have 分類
  ↓
【ステップ 2】対応判定
  - Critical: 100% 対応 ✅
  - Important: 品質スコア + 実装コスト で判定
  - Nice-to-have: 効率判断で選別
  ↓
【ステップ 3】実装 + テスト
  ↓
【ステップ 4】品質スコア を記録
  ↓
このダッシュボードに記載
```

## 📝 使用方法

### PR ごとのメトリクス記録

PR マージ直後に以下を実行：

```bash
# PR#29 の例
# 1. 上の表に新行を追加
# 2. コメント数・分類を記載
# 3. 対応率を計算（対応したコメント数 / 総数）
# 4. 品質スコアを主観的に評価
# 5. このファイルをコミット
git add docs/agent-performance.md
git commit -m "docs: Agent パフォーマンス記録 PR#XX"
```

### 月次集計

毎月末に：
1. 全 PR のメトリクスを集計
2. 対応率・品質スコア の平均を更新
3. トレンド分析を実施
4. 改善機会を特定

### PR #27 評価詳細

**PR**: 店舗設定 API（バックエンド）  
**Issue**: #10  
**評価日**: 2026-04-12

#### コメント分類と対応

| コメント # | 内容 | 分類 | 対応 | 評価スコア |
|----------|------|------|------|----------|
| 1 | getSettings() で findOne の非決定性 | 🔴 Critical | ✅ 対応 | ⭐⭐⭐⭐⭐ |
| 2 | updateSettings() の検証ロジック漏れ | 🔴 Critical | ✅ 対応 | ⭐⭐⭐⭐⭐ |
| 3 | JWT ペイロードが複雑 | 🔴 Critical | ✅ 対応 | ⭐⭐⭐⭐ |
| 4 | Swagger デコレータ不足 | 🔴 Critical | ✅ 対応 | ⭐⭐⭐⭐⭐ |
| 5-7 | 4 コメント（Important 分類） | 🟡 Important | ✅ 全対応 | ⭐⭐⭐ |
| 8-11 | 3 コメント（Nice-to-have） | 🟡 Nice-to-have | ❌ 非対応 | ⭐⭐⭐ |

#### 品質評価結果

| 評価項目 | スコア | 備考 |
|---------|--------|------|
| 精度 | 85% | Critical は全て的確。Nice-to-have に軽微なもの混在 |
| 価値 | 80% | 機能問題の指摘は高価値。スタイル指摘は優先度判定で選別可 |
| 完全性 | 75% | 型安全性など重要な指摘あり。一部重複あり |
| **総合品質** | **⭐⭐⭐⭐** | **良好 (3.5/5)** |

#### Agent の理解度
- 実装内容の理解: ⭐⭐⭐⭐ (大部分理解、詳細は推測)
- コード品質への洞察: ⭐⭐⭐ (重要な問題を指摘、重複あり)
- 提案の実装可能性: ⭐⭐⭐⭐ (大部分は即座に使える)

#### 結論

Critical コメントの精度は高いが、全 11 コメント中 7 コメントが非対応となった。原因は：
1. Nice-to-have に軽微な改善が多かった
2. 実装コストが高い割に効果が薄い指摘があった
3. 優先度判定により効率重視で選別

今後は指摘の優先度分類をさらに正確にすることで、対応率向上が期待できる。
